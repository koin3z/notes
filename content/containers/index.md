## コンテナとは

コンテナは、アプリケーションとその実行に必要なファイル、ライブラリ、設定などをまとめ、ホストOS上で分離して実行するための仕組み。

仮想マシンのようにゲストOS全体を起動するのではなく、ホストOSのカーネルを共有しながら、プロセス・ファイルシステム・ネットワークなどを隔離する。実態としてはホスト上のプロセスだが、コンテナ内から見ると独立した実行環境のように見える。

- アプリケーションと依存関係をまとめて配布できる
- 実行環境の差異による不具合を減らせる
- 仮想マシンより起動が速く、CPUやメモリのオーバーヘッドが小さい
- LinuxコンテナはLinuxカーネルを前提に動く
- macOSやWindowsでLinuxコンテナを動かす場合、多くは内部でLinux VMを利用する

## コンテナを支える仕組み

Linuxコンテナは、主に次のカーネル機能によって成り立っている。

| 機能                         | 役割                                                                 |
| ---------------------------- | -------------------------------------------------------------------- |
| namespaces                   | プロセスID、ネットワーク、マウント、ホスト名などの見え方を分離する   |
| cgroups                      | CPU、メモリ、I/Oなどのリソース利用量を制限・管理する                 |
| capabilities                 | root権限を細かい権限単位に分割し、不要な権限を落とす                 |
| seccomp / AppArmor / SELinux | システムコールやアクセス権を制限し、攻撃面を小さくする               |
| UnionFS / OverlayFS          | イメージの読み取り専用レイヤと、コンテナごとの書き込みレイヤを重ねる |

コンテナは「小さなOS」そのものではなく、隔離されたユーザーランドを持つプロセスまたはプロセス群と捉えると理解しやすい。

### ユーザーランド

ユーザーランドは、カーネルの外側で動くコマンド、シェル、ライブラリ、設定ファイル、アプリケーションなどの領域を指す。コンテナはこのユーザーランドをイメージとして持つため、Ubuntu、Alpine、Debianなどの環境がコンテナ内にあるように見える。

ただし、カーネルはホスト側のものを共有する。たとえばUbuntuベースのコンテナを起動しても、コンテナごとにUbuntuのカーネルが起動しているわけではない。

## 仮想マシンとの違い

仮想マシンは、ハイパーバイザを通して仮想的なハードウェアを提供し、その上でゲストOSを起動する。一方、コンテナはホストOSのカーネルを共有し、OSレベルでプロセスを分離する。

| 観点         | コンテナ                                      | 仮想マシン                           |
| ------------ | --------------------------------------------- | ------------------------------------ |
| 分離の単位   | プロセス、ユーザーランド                      | マシン、ゲストOS                     |
| カーネル     | ホストOSのカーネルを共有                      | ゲストOSごとに持つ                   |
| 起動速度     | 速い                                          | 比較的遅い                           |
| リソース消費 | 少ない                                        | 多い                                 |
| 隔離の強さ   | カーネル共有のためVMより弱い                  | OSごと分離するため強い               |
| 主な用途     | アプリケーション実行、CI/CD、マイクロサービス | 異なるOSの実行、強い分離が必要な環境 |

コンテナは軽量で扱いやすい反面、カーネルを共有するため、隔離境界としては仮想マシンより薄い。強い分離が必要な場合は、VMやKata ContainersのようなVMベースのコンテナ実行基盤を検討する。

## コンテナ型仮想化

コンテナ型仮想化は、OSレベルの仮想化によって複数の分離されたユーザー空間を提供する方式。代表的な実装やツールには次のものがある。

- Docker
- containerd
- Podman
- LXC / LXD
- systemd-nspawn
- Solaris Zones

Dockerはコンテナ技術を広く普及させた代表的なツールだが、コンテナそのものはDocker専用の概念ではない。現在はOCI（Open Container Initiative）によって、コンテナイメージやランタイムの仕様が標準化されている。

## アプリケーションコンテナとシステムコンテナ

コンテナには、大きく分けて「アプリケーションコンテナ」と「システムコンテナ」という考え方がある。どちらも namespaces や cgroups など同じLinuxカーネル機能を使うが、何を1つのコンテナとして扱うかが異なる。

| 観点           | アプリケーションコンテナ               | システムコンテナ                     |
| -------------- | -------------------------------------- | ------------------------------------ |
| 思想           | 1プロセスまたは1サービスを実行する単位 | 軽量なLinux環境を実行する単位        |
| PID 1          | アプリケーション本体やエントリポイント | systemdなどのinit                    |
| プロセス数     | 基本は少数                             | 複数のデーモンやサービスを実行できる |
| ライフサイクル | 短命・使い捨てにしやすい               | 長期稼働させやすい                   |
| 状態           | ステートレスに寄せる                   | ステートフルになりやすい             |
| 代表ツール     | Docker、containerd、CRI-O、Podman      | LXC / LXD、systemd-nspawn            |

### アプリケーションコンテナ

アプリケーションコンテナは、特定のアプリケーションやサービスを実行するための単位。DockerやKubernetesで一般的に使われるモデル。

- コンテナ起動時に指定したメインプロセスがPID 1になる
- メインプロセスが終了すると、通常はコンテナも終了する
- SSHログインや常駐デーモンを前提にせず、ログは標準出力・標準エラーに出す
- 設定変更や更新は、コンテナ内を直接変更するよりイメージの再ビルド・再デプロイで行う
- 状態はコンテナ外のボリューム、データベース、オブジェクトストレージなどに逃がす

「1コンテナ=1プロセス」と説明されることが多いが、厳密には1つのコンテナ内で複数プロセスを動かすこともできる。重要なのは、1つの主要な責務を持つ実行単位として扱うこと。

### システムコンテナ

システムコンテナは、仮想マシンに近い「軽量なLinux環境」を提供するモデル。1つのコンテナ内でinitが動き、その配下で複数のサービスを実行できる。

- PID 1としてsystemdなどのinitが動く
- SSHログイン、パッケージ追加、複数デーモンの起動といった運用がしやすい
- VMのように長期間稼働させる用途に向いている
- 代表例はLXC / LXD、systemd-nspawn、OpenVZ

アプリケーション単位でのデプロイよりも、開発環境、検証環境、軽量なサーバー環境のような用途に向いている。

## LXC と LXD

### LXC

LXC（Linux Containers）は、Linuxのコンテナ機能を使ってシステムコンテナを作成・実行するための低レベル寄りのツール群。Linux上に、独立したLinux環境を作る用途で使われる。

- systemdなどを含むフルに近いLinuxユーザーランドを動かせる
- Dockerのようなアプリケーションパッケージングより、軽量VMに近い使い方をする
- ネットワーク、ストレージ、権限、プロファイルなどを細かく設定できる

### LXD

LXDは、LXCを扱いやすくするためのコンテナ・仮想マシンマネージャー。システムコンテナだけでなく、仮想マシンも管理できる。

- 事前に用意されたイメージからLinux環境を作成できる
- REST APIとCLIでコンテナやVMを管理できる
- プロジェクト、プロファイル、スナップショット、イメージ管理などの運用機能を持つ
- Ubuntuを中心に使われることが多いが、複数のLinuxディストリビューションのイメージを扱える

Dockerが「アプリケーションをイメージ化して実行する」ことに強いのに対し、LXDは「Linux環境を軽量に作成して管理する」ことに強い。

## メリット

- 実行環境をイメージとして固定できる
- 開発、検証、本番の環境差分を減らせる
- 起動・停止が速く、スケールアウトしやすい
- イメージをレジストリ経由で配布・共有できる
- CI/CDやKubernetesなどの自動化基盤と相性がよい

## 注意点

- カーネルを共有するため、カーネル脆弱性の影響を受ける可能性がある
- コンテナ内でrootを使う場合、ホスト側への影響を考慮する必要がある
- コンテナ数が増えると、ログ、監視、ネットワーク、権限管理が複雑になる
- イメージに脆弱なパッケージや秘密情報を含めないようにする必要がある
- ステートフルなデータをコンテナ内だけに置くと、再作成時に失われる
- LinuxコンテナとWindowsコンテナは前提とするカーネルが異なるため、同じものとして扱えない

## コンテナとは

- プログラムやライブラリの依存関係をまるっと含めた，実行環境を閉じ込めた環境で管理されるプロセス
	- アプリケーションをコンテナごとに管理することで，実行環境ごと移動させることができる
	- デプロイ先やバージョン違いによるエラーを回避できる
- シェルやコマンド，ライブラリ，ファイルシステムなど，OSの要素のうちのツールを指すユーザーランドをコンテナは個別にもつ
- その一方で，Linuxカーネルはコンテナで共有して使用する
	- ゲストOSがあるように見えるのは，このLinuxカーネルを共有しているため
- そのため実行環境には，Linuxカーネルを用意する必要がある
- 一方で仮想マシンは，Linuxカーネルごと仮想化する点が異なる
- ユーザーランドはカーネルの外の要素全体を指し，カーネルから割り当てられたuser spaceで実行される
- 複数のユーザー空間を使用してプロセスを分離，単一のOSを実行する

- 実態としてはホストOSのプロセス
	- そのため，CPUやメモリのリソース消費が少ない
	- 展開が高速


## 仮想マシンとの違い
- ホストOS（またはハイパーバイザ）上で演算処理により仮想的なマシン環境を作り、その上でOSを実行させる仮想化技術がある
	- Xen や KVM など
- この場合、ホストOSとゲストOSを完全に分離できるというメリットがあるが、ハードウェアを演算によってシミュレートするため、そのためのオーバーヘッドが必要になるデメリットがある

## コンテナのデメリット

- Linuxベースのディストリビューションは広く展開されているが，Windowsは十分でない
- カーネルを共有するため，カーネルの脆弱性によってすべてのコンテナがリスクがさらされる可能性がある
- 複数コンテナを実行した場合，数多くのプロセスが走ることになり，監視することが難しい

## コンテナ型仮想化

- コンテナごとにカーネルのプロセス管理やコンテナの隔離，及びインターフェースを提供するソフトウェア
- コンテナ，という用語は「OSレベルの仮想化（OS-level virtualization）」のシステムを指すことが多い
- 

- 主要なものとしてDockerがあるがその他に
	- LXC
	- Solaris containers
	- Podman
- などがある

- 複数の分離したユーザー空間インスタンスをカーネルによって割り当てる。
- コンテナ内で動作する

## 「アプリケーションコンテナ」と「システムコンテナ」
- 基本的にコンテナは単一のプロセス
	- 仮想環境というよりは、単一のアプリケーションパッケージに近い
- だが、軽量なVMとしてのLinuxコンテナを扱うこともできる
	- → システムコンテナ
		- LXD
		- LXC
### 比較表

|観点|アプリケーションコンテナ|システムコンテナ|
|---|---|---|
|思想|1プロセス／サービス|軽量OS環境|
|PID 1|アプリ本体|init（systemd等）|
|ライフサイクル|短命・使い捨て|長期稼働|
|状態|ステートレス|ステートフル|
|運用比喩|家畜（cattle）|ペット（pet）|
|代表ツール|Docker, containerd|LXC/LXD, systemd-nspawn|
### 技術的には同じ基盤

どちらも Linux の **namespace**（プロセス・ネットワーク・マウント等の分離）と **cgroups**（リソース制限）という同じカーネル機能の上に成り立っています。違いは技術的な隔離方式ではなく、**「何を1つのコンテナとして扱うか」という設計哲学・運用パターン** にあります。

補足として、近年はこの境界が曖昧になりつつあります。Kubernetes の Pod は複数コンテナをまとめる点でシステムコンテナ的な側面を持ちますし、Kata Containers のように VM レベルの隔離をアプリケーションコンテナのワークフローに組み込む技術も出てきています。

### アプリケーションコンテナ
**1コンテナ＝1プロセス（1サービス）** を基本思想とするモデルです。Docker が普及させた考え方で、コンテナは特定のアプリケーションプロセスを実行するための使い捨ての単位です。

- 起動すると PID 1 として単一のメインプロセスが走り、そのプロセスが終了するとコンテナも終了する
- OS の init システム（systemd など）やログイン、cron、SSH デーモンは持たない
- 状態を持たず（stateless）、イミュータブルに扱われる。設定変更は再ビルド・再デプロイで行う
- 代表例: Docker、containerd、CRI-O（Kubernetes が利用）

### システムコンテナ
**仮想マシンに近い「軽量なOS環境」** を提供するモデルです。1つのコンテナの中で init システムが動き、複数のプロセスやサービスを同時に走らせます。

- PID 1 に systemd などの init が入り、その配下で複数のデーモンが動く
- SSH ログインしてパッケージを入れたり、長期間稼働させたりと、永続的（persistent）に運用できる
- VM のように「箱」として扱われ、ペット的（手をかけて世話をする）な運用になりやすい
- 代表例: LXC / LXD、systemd-nspawn、OpenVZ
### LXD 
- コンテナ＆仮想マシンマネージャー
	- Ubuntu 開発を支援する Canonical社によって作成
	- ホストOSとしては Ubuntu がおすすめ
- 軽量な Linux 環境を高速に構築
	- Docker と同様にあらかじめビルドされたコンテナイメージを展開する
- また、コンテナだけでなく、仮想マシンを管理することもできる

- 一方、Dockerとは異なりOSのフル機能が使用できる
	- コンテナの手軽さで使えるVMのような環境

## PQCとは

Post-Quantum Cryptography（PQC）は、量子コンピューターを使う攻撃者に対しても安全性を維持できると
考えられる、古典コンピューター上で動作する暗号方式である。量子鍵配送（QKD）とは別の技術である。

## 量子計算による脅威

### Shorのアルゴリズム

十分に大規模で誤り訂正された量子コンピューターが実現すると、素因数分解と離散対数に依存する RSA、DH、
ECDH、ECDSA、EdDSA などの公開鍵暗号へ根本的な影響を与える。単に鍵長を増やす対策では不十分となる。

### Groverのアルゴリズム

総当たり探索を平方根程度に高速化し、対称鍵暗号やハッシュの原像探索へ影響する。公開鍵暗号に対する
Shor の影響と同じではなく、十分な鍵長・出力長の選択で安全余裕を確保する考え方がある。

### Harvest Now, Decrypt Later

攻撃者が現在の暗号通信を保存し、将来の量子コンピューターで復号する脅威。医療、政府、知的財産、
長期契約など、長い機密保持期間を持つデータは、量子コンピューター完成前から移行判断が必要になる。

## NISTの標準

2024年8月13日に最初の3標準が確定した。

| FIPS     | アルゴリズム | 用途                 | 主な基礎                             |
| -------- | ------------ | -------------------- | ------------------------------------ |
| FIPS 203 | ML-KEM       | 鍵カプセル化・鍵確立 | Module-LWE 系の格子問題              |
| FIPS 204 | ML-DSA       | デジタル署名         | Module-LWE / Module-SIS 系の格子問題 |
| FIPS 205 | SLH-DSA      | デジタル署名         | ハッシュ関数                         |

ML-KEM は一般用途の鍵確立で中心となる。ML-DSA は一般用途の署名、SLH-DSA は異なる安全性仮定を持つ
バックアップ的な選択肢となる。名称は候補時代の Kyber、Dilithium、SPHINCS+ と標準名を混同しない。

## 標準化中の方式

- FN-DSA（FALCON 由来）は FIPS 206 として開発中
- HQC は2025年に ML-KEM のバックアップとなる KEM として選定され、標準化作業中
- 追加のデジタル署名方式も NIST の評価プロセスが継続中

2026年7月1日時点では、これらを FIPS 203〜205 と同じ「確定済み標準」として扱わない。実装前に NIST の
最新ステータスを確認する。

## 数学的ファミリー

- **格子ベース**: 性能とサイズのバランスがよく、ML-KEM・ML-DSA の基礎
- **ハッシュベース**: ハッシュ関数への依存が中心で保守的だが、署名サイズなどにトレードオフ
- **符号ベース**: 誤り訂正符号の復号困難性を利用。HQC など
- **多変数多項式・同種写像など**: 研究・評価が続く。候補の破綻例もあり、方式分散が重要

「量子耐性」は将来にわたる無条件の安全保証ではない。新しい暗号解析、実装不備、サイドチャネル、乱数、
プロトコル統合を継続評価する。

## 移行はアルゴリズム置換だけではない

PQC は公開鍵や署名が大きくなる場合があり、通信、証明書、HSM、ファームウェア、プロトコル、ログ、
データベース列長、性能、可用性へ影響する。既存クライアントとの相互運用やロールバックも必要となる。

### 1. 暗号資産を棚卸しする

- TLS、VPN、SSH、PKI、証明書、コード署名、文書署名
- KMS、HSM、秘密管理、バックアップ暗号化
- アプリケーション独自暗号、組込み機器、長寿命製品
- アルゴリズム、鍵長、ライブラリ、プロトコル、所有者、更新可否
- データの機密保持期間とシステム寿命

### 2. 優先順位を付ける

`移行に必要な期間 + データを秘密に保つ期間` が、量子脅威が現実化するまでの想定期間を超える領域を
優先する。外部公開、長期署名検証、更新困難な機器、重要インフラも考慮する。

### 3. Crypto-agilityを高める

- アルゴリズムや鍵をコードへ固定しない
- 暗号ポリシーと実装を分離する
- バージョン交渉、鍵・証明書ローテーション、失効を自動化する
- ハイブリッド方式を採る場合は、標準・脅威モデル・結合方法を確認する
- 相互運用、性能、障害、ダウングレード攻撃をテストする

### 4. 段階移行する

検証環境、限定ワークロード、デュアルスタック、本番移行、従来方式廃止の順に進める。ベンダーのロードマップ、
規制・調達要件、[[memos/fips|FIPS 140-3]] 検証状況も追跡する。

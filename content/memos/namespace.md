## 1. Namespace とは何か

Namespace（名前空間）は、Linuxカーネルの機能で、**プロセスが見えるシステムリソースの「ビュー」を分離する**仕組みです。

同じカーネル上で動作していても、異なるnamespaceに属するプロセスは、それぞれ「自分だけの独立した環境」を持っているかのように振る舞います。

> **重要な原則**：Namespace は「見え方の分離」であり、VMのような完全なカーネル分離ではない。

---

## 2. chroot との関係

### chroot とは

1979年に登場した古い機能。プロセスの**ルートディレクトリを付け替えるだけ**の仕組み。

```
ホストのファイルシステム
/
├── bin/
├── etc/
└── jail/          ← ここを新しい「/」に見せる
    ├── bin/
    └── etc/
```

### chroot の限界：脱獄（Breakout）

```c
// chroot breakout の古典的パターン（概念）
mkdir("tmpdir");
chroot("tmpdir");        // 偽のchrootに入る
for (int i = 0; i < 256; i++)
    chdir("..");         // 本物の "/" まで上る
chroot(".");             // 本物のルートを奪取
```

**なぜ効くのか？** chroot はルートディレクトリを差し替えるだけで、現在の作業ディレクトリや既に開いているファイルディスクリプタまでは閉じないため。ファイルシステムの見え方を変えるだけで、完全なサンドボックスではない。

### chroot vs MNT Namespace

|観点|chroot|MNT namespace|
|---|---|---|
|分離対象|ルートディレクトリのみ|マウントポイント全体|
|登場年|1979|2002|
|脱獄の難しさ|**容易**（root権限で脱獄可能）|より困難|
|カーネルレベル|ルートポインタの付け替えのみ|マウントテーブル自体を分離|

```
カーネル内部

chroot:
  プロセスA ─── 共通マウントテーブル（ホストと同じ）
                └── ルートポインタだけ差し替え

MNT namespace:
  プロセスA ─── マウントテーブルA（コピー・独立）
  プロセスB ─── マウントテーブルB（完全に別物）
```

## 3. 主要な Namespace の種類

|Namespace|分離対象|コンテナでの役割|
|---|---|---|
|**PID**|プロセスID|コンテナ内でPID 1から始まる独自のプロセスツリー|
|**MNT**|ファイルシステムのマウント|独自のファイルシステムビュー|
|**NET**|ネットワークスタック|独自のNIC・IPアドレス|
|**USER**|ユーザー・グループID|UID/GIDのマッピング|
|**UTS**|ホスト名・ドメイン名|コンテナ固有のホスト名|
|**IPC**|プロセス間通信リソース|独立したIPCオブジェクト|
|**CGROUP**|cgroupのルート|cgroup階層の見え方を分離|

## 4. PID Namespace

### 挙動

コンテナ内の最初のプロセスは **PID 1** として見える。ホスト側では別のPIDが割り当てられている。

```
ホスト側          コンテナ内
─────────────────────────────
PID 1 (init)
PID 2 (kthread)
...
PID 501 (bash)  ──→  PID 1 (bash)   ← コンテナ内では PID 1 に見える
PID 502 (nginx) ──→  PID 2 (nginx)
```

### 分離と保護の違い

```
コンテナ内 PID 1 が死ぬ
  └── コンテナのみ終了　→ ホストは無影響

コンテナから kill PID 1
  └── コンテナ内の PID 1 = ホストの PID 501
      → ホストの PID 501 が終了するだけ
      → ホストの本物の PID 1 には届かない
```

**ポイント**：PID namespace はプロセスの「見え方」を分離するだけでなく、ホスト側のプロセスツリーへのアクセスを遮断している。

## 5. MNT Namespace

### 挙動

コンテナ内のファイルシステムはイメージのレイヤーから構成され、ホストの `/bin` や `/etc` とは完全に別物。

```
コンテナ内            実体
─────────────────────────────────────
/bin                ← イメージのレイヤー（独立）
/etc                ← イメージのレイヤー（独立）
/var                ← イメージのレイヤー（独立）
```

### ボリュームマウント（-v オプション）

```bash
docker run -v /host/data:/container/data ubuntu
```

ホストとコンテナで**同じ inode（実体）を共有**する。コピーではなく、同じものを別の名前で見ている状態。

```
ホスト                    コンテナ
/host/data/file.txt  ←→  /container/data/file.txt
        ↑
    同じ inode（実体は一つ）
```

### 危険なマウントの例

```bash
# ホストのルートファイルシステム全体をマウント
docker run -v /:/host ubuntu
→ コンテナ内からホスト全体が丸見え・改ざん可能

# ホストのetcをマウント
docker run -v /etc:/host-etc ubuntu
→ /host-etc の削除・改ざんがホストに直撃

# Docker ソケットをマウント（最も危険）
docker run -v /var/run/docker.sock:/var/run/docker.sock ubuntu
→ コンテナからDockerデーモン（root権限）に命令できる
```

### Docker ソケットマウントのリスク

Docker ソケット（`/var/run/docker.sock`）はDockerデーモンへの通信口。これをマウントすると：

```
攻撃者がコンテナ内から実行できること：

1. 別コンテナの制御
   └── 停止・削除・内部への侵入

2. ホストファイルシステムの奪取
   docker run -v /:/host ubuntu
   └── ホスト全体がコンテナから丸見え・改ざん可能

3. 特権コンテナの起動
   docker run --privileged ubuntu
   └── capability・device・LSM などの制限が大きく緩む
```

## 6. NET Namespace

### 挙動

コンテナ内では、コンテナに許可された NIC だけが見える。

```
コンテナ内 ip addr
─────────────────────────────
lo        127.0.0.1      ← ループバック
eth0      172.17.0.2     ← コンテナ専用の仮想NIC

ホストの eth0（192.168.1.x）は見えない
```

### veth ペアと NET Namespace の関係

NET namespace は「ネットワークスタックの分離」を担い、veth はその「分離された空間を繋ぐ配線」。

veth ペアは**2つの異なる NET namespace をまたいで接続する用途で典型的に使われる**仮想デバイス。

```
NET namespace A (コンテナ)    NET namespace B (ホスト)
──────────────────────────    ────────────────────────
eth0 ←───────────────────────────→ veth0
                                        │
                                    docker0 (bridge)
                                        │
                                    ホスト eth0
```

### --network=host のリスク

```bash
docker run --network=host ubuntu
```

NET namespace を作らず、ホストの namespace をそのまま使う。veth すら不要でホストのネットワークスタックに直接乗る。

```
通常のコンテナ
  NET namespace (コンテナ) ←veth→ NET namespace (ホスト)
  └── 分離されている

--network=host
  NET namespace を作らず、ホストの namespace をそのまま使う
  └── ネットワーク面の隔離がなくなる
      ※ 他の namespace 分離まで消えるわけではない
```

## 7. USER Namespace

### UID マッピングの仕組み

コンテナ内で `root`（UID 0）として動くプロセスを、ホスト側では非特権ユーザーにマッピングできる。

```
コンテナ内          ホスト側
UID 0 (root)  ──→  UID 100000 (一般ユーザー)
```

マッピングは**オペレーター（ホスト側のユーザーや設定ファイル）が定義**し、カーネルが執行する。

```
/etc/subuid (ホストの設定ファイル)
─────────────────────────────────
alice:100000:65536
└── alice というユーザーが
    コンテナ内 UID 0 を ホスト UID 100000 にマッピングできる
```

### chroot 脱獄との組み合わせ

```
chroot 脱獄に成功してホストの / に到達しても…
  └── ホストから見れば UID 100000（一般ユーザー）として識別される
        → ホストの重要なリソースへのアクセス権限がない
```

chroot 単体の脱獄が「root のまま外に出る」ことを前提としていたのに対し、USER namespace がその前提を崩している。

### マッピング範囲外の UID

```
マッピング定義された範囲内
  コンテナ内 UID 0  ──→  ホスト UID 100000
  └── ホスト上では一般ユーザーとして動く

マッピング範囲外の UID がユーザー空間に見える場合
  ──→  overflow UID/GID（多くの環境では 65534）として表示される
  └── ホスト上の実権限を得られるわけではない
```

カーネルはマッピングテーブルを**信頼の境界として機能**させている。

## 8. コンテナセキュリティの全体像

### 技術の積み重ね

```
技術              役割                        限界
──────────────────────────────────────────────────────────
chroot            ファイルシステムの見た目を変える   root で脱獄可能
MNT namespace     マウントテーブル自体を分離
USER namespace    UID マッピングで権限を無効化
PID namespace     プロセスの見え方を分離
NET namespace     ネットワークスタックの分離
seccomp           システムコールを制限
cgroup            CPU/メモリなどリソース使用量を制限
──────────────────────────────────────────────────────────
共通の限界 → カーネルを共有しているため、
            カーネルの脆弱性はすべてのコンテナに影響する
```

### コンテナ vs VM の本質的な違い

```
VM:
  ハードウェア
  └── Hypervisor
      ├── カーネルA (VM1)
      └── カーネルB (VM2)  ← カーネルが分離

コンテナ:
  ハードウェア
  └── カーネル（共有）
      ├── namespace A (コンテナ1)
      └── namespace B (コンテナ2)  ← カーネルは一つ
```

### システムコールとカーネル共有の問題

```
PID namespace
  └── プロセスの「見え方」を分離　→ 分離 ✅
        でも…
  └── システムコールは共有カーネルで処理　→ 保護 ❌

つまり：
コンテナ内の悪意あるプロセス
  └── 脆弱なシステムコールを呼び出す
        └── ホストカーネルで処理される
              └── カーネル自体を攻撃できる可能性
```

これが **カーネルエクスプロイト** がコンテナセキュリティにおいて根本的な脅威である理由。

## 9. cgroup と cgroup Namespace の違い

### cgroup（リソース制限）とは

CPU・メモリ・I/O・プロセス数などのリソース使用量を**制限・計測する仕組み**。namespace とは独立した別の Linux カーネル機能。

```
cgroup の役割：
  コンテナA → CPU 最大 2コア、メモリ 512MB まで
  コンテナB → CPU 最大 1コア、メモリ 256MB まで
```

### cgroup Namespace とは

cgroup という仕組みの「**見え方**」を分離する namespace。

```
cgroup namespace がない場合：
  コンテナ内から /proc/self/cgroup を見ると…
  └── ホスト側の cgroup パスが見えることがある
        ├── ランタイムや slice 名などのホスト側構造を推測できる
        └── 他の階層名が見える可能性

cgroup namespace がある場合：
  コンテナ内から /proc/self/cgroup を見ると…
  └── 自分のルートが / に見える（切り詰められた view）
        └── ホスト側の cgroup パスが隠蔽される
```

### 二つの関係

```
cgroup（リソース制限）
  └── CPU・メモリなどの使用量を実際に制限する仕組み

cgroup namespace
  └── その cgroup 階層の「見え方」を分離する namespace
        └── コンテナ内からホスト側の cgroup パスを見えにくくする
```

つまり **cgroup namespace は cgroup という仕組みの上に乗っている**。リソースを制限するのが cgroup、その情報の見え方を隠蔽するのが cgroup namespace。

### 他の namespace との対比

```
namespace       何の「見え方」を分離するか    隠蔽される情報
──────────────────────────────────────────────────────
PID             プロセスツリー               ホストのプロセス数・構成
NET             ネットワークスタック          ホストのNIC・IP構成
MNT             ファイルシステム             ホストのディレクトリ構造
USER            UID/GID                     ホストのユーザー構成
cgroup          cgroupの階層                ホストのリソース規模・他コンテナの存在
```

## 10. 攻撃面と対策のまとめ

|攻撃面|対策技術|
|---|---|
|ファイルシステムへのアクセス|MNT namespace|
|root 権限の悪用|USER namespace（UID マッピング）|
|ホストプロセスへの干渉|PID namespace|
|ネットワークの盗聴・なりすまし|NET namespace|
|カーネルへの直接操作|seccomp|
|リソースの枯渇|cgroup|

> **根本的な限界**：カーネルを共有しているため、カーネル自体の脆弱性はすべてのコンテナに影響する。この限界を解決しようとしているのが **gVisor** や **Kata Containers** などの技術。

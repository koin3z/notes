#### Zero Data Loss Autonomous Recovery Service

- RCV/ZRCVの内部ストレージは「デルタストア（Delta Store）」と呼ばれ，Oracleブロックを認識して管理。
- 単にデータを保存するだけでなく，バックアップデータが到着した瞬間や，保管されている間（At Rest）に，継続的な整合性チェック（バリデーション）が実行される。
- これにより，「バックアップは成功していたが，リストアしようとしたらデータが破損していて戻せなかった」という，バックアップ運用における最悪の事態を未然に防止する。
- 常に「復元可能であること」をシステム的に保証し続ける 。

## RCV
自立型リカバリーサービス

- オンプレミスで提供される「Zero Data Loss Recovery Appliance」の機能が，クラウドでも利用できるもの
- 増分更新バックアップを行うが，フルバックアップへのマージを行わない
- RCVの中で独自に増分バックアップが格納され，仮想的なフルバックアップが作れる仕組みになっている

- 仮想的なフルバックアップが作成できるので，リストア時の増分バックアップ分のリストア（マージ）が不要に
- また，マージをしないので，過去の時点に戻すことも可能になっている

- OSからはバックアップの領域が見えないので，ランサムウェア対策としても有効
- RCVは，定期的なアーカイブログのバックアップに依存。
	- 例えば，アーカイブログのバックアップ間隔が15分に設定されている場合，障害発生時点によっては最大15分間のデータ（最後にバックアップされた時点から障害発生までの間のトランザクション）が失われるリスクが。
	- これは従来のRMANによるバックアップ運用と同様のリスクプロファイル 。

## ZRCV
- リアルタイムデータ保護，というチェック項目を有効化するとZRCVになる
- Zero Data Loss RCV (ZRCV)
- オンラインREDOログもバックアップされるようになり，データが欠けることがない
- ZDLRAの技術的系譜を直接受け継ぎ，データベースのREDOログ（トランザクション履歴）がメモリ上でコミットされると同時に，リアルタイムでリカバリサービスへ転送・保護
- ZRCVの最大の特徴は，1秒未満の目標復旧時点（Sub-second RPO） の実現。
	- 障害や攻撃が発生した際，最後にコミットされたトランザクションまでのデータを完全に復旧可能にすることで，「データ損失ゼロ（Zero Data Loss）」を保証する 。
- ZRCVは，データベース内部のログライター（LGWR）プロセスと連携し，リアルタイムREDO転送を行う。
	- データベース上でトランザクションがコミットされると，その変更履歴（REDOエントリ）は即座にリカバリサービスへと送信されます。これはOracle Data Guardがスタンバイデータベースへログを送る仕組みと似ているが，宛先がリカバリサービスである点が異なる。
	- このメカニズムにより，仮にデータベースサーバーが物理的に全損（火災やハードウェアの溶解など）した場合でも，最後にコミットされたトランザクションデータはすでにZRCV側に安全に保管され，データの欠陥を防ぐ。

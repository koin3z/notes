## 概要
- OAuthフローにおいて，認可サーバーが行う役割は以下の通り
	- ユーザー認証
	- クライアントの登録
	- トークンの発行
- 認可サーバーは２つのエンドポイントを持つ
	- 認可エンドポイント
	- トークンエンドポイント

## todo
### クライアントの登録
- 認可サーバーはクライアントについて一意に識別するための情報，`client_id`を発行する
	- この登録方法については「静的」と「動的」の２種類がある
	- 機密クライアントの場合，`client_secret`も発行する
- また，認可コードフローを使用する場合，クライアントから認証後のリダイレクト先として`redirect_uri`を登録してもらう必要がある
	- この`redirect_uri`は複数もたせることができる
	- これはさまざまな状況で異なるURLを処理するようなアプリケーションで使用することを想定しているためで，URLごとにリソースオーナーの同意内容を紐づけることができる

```javascript
let client = [
	{
		"client_id": "xxxxxxxxxxx",
		"client_secret": "yyyyyyyyyyy",
		"redirect_uris":["https://zzzzzzzzzz/callback"]
	}
]
```


### 認可エンドポイントでの処理

- 認可エンドポイント
	- クライアントがリソースオーナーのブラウザを使用したリダイレクトでリクエストを送るエンドポイント
- ここへのリクエストは常にGETを使用する

- 認可エンドポイントでは，どのクライアントからリクエストが行われているかを識別し，そのクライアントが存在しているかを確認
	- 識別子としては`client_id`を利用する
	- また，フロントチャネルを利用するため，クライアントの偽装対策として正当性を確認する必要がある
		- 正当性の確認要素として，`redirect_uri`の比較がある

- クライアントの正当性を確認次第，リソースオーナーを認証し，クライアントに対して権限を許可するかを聞く

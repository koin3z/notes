## 統制とは

統制（control）は、リスクを低減し、要求事項を満たすための措置である。製品機能だけでなく、
方針、承認、教育、監視、レビューなども含む。

| 分類       | 目的                                               | 例                                         |
| ---------- | -------------------------------------------------- | ------------------------------------------ |
| 予防的統制 | 事象を起こりにくくする                             | MFA、最小権限、変更承認、セキュア設定      |
| 発見的統制 | 発生した事象を見つける                             | ログ監視、アラート、棚卸し、脆弱性スキャン |
| 是正的統制 | 影響を抑え、正常状態へ戻す                         | 隔離、パッチ、バックアップからの復旧       |
| 抑止的統制 | 不正行為を思いとどまらせる                         | 利用規程、警告表示、懲戒規定               |
| 補完統制   | 本来の統制を実装できない場合に同等の目的を達成する | ネットワーク分離と監視の強化               |

## 統制記述に必要な要素

曖昧な「適切に管理する」では、実施も監査もできない。最低限、次を記述する。

- **目的**: どのリスクまたは要求を扱うか
- **対象**: システム、データ、利用者、環境の範囲
- **責任者**: 統制オーナーと実施担当
- **活動**: 何をどの手順で行うか
- **頻度・契機**: 毎日、四半期、入退社時、変更時など
- **証跡**: 実施結果を何で示すか
- **例外**: 誰が期限付きで承認し、どの補完統制を置くか
- **評価方法**: 設計と運用有効性をどう確認するか

例: 「情報システム部門は、四半期ごとに本番環境の特権アカウント一覧を人事記録と照合し、
システム所有者の承認を得る。差異はチケットで追跡し、一覧、承認記録、是正チケットを1年間保管する。」

## 証跡の品質

良い証跡は、対象期間の統制が実際に動いたことを第三者が再確認できる。

- 誰が、いつ、何を実施・承認したかが分かる
- 対象母集団と抽出条件が分かる
- 元データとの対応が取れ、改ざんや上書きを検知できる
- 例外、失敗、再実施を成功例と一緒に残している
- 対象期間と保存期間が要求に合う
- タイムゾーン、システム時刻、ログ欠損を説明できる

スクリーンショットだけでは、検索条件、母集団、取得者、取得日時が欠けやすい。可能なら、API出力、
監査ログ、承認履歴、設定の構成管理記録など、再現可能な一次データを残す。

## 設計有効性と運用有効性

- **設計有効性**: 統制が目的を達成できるよう設計されているか
- **運用有効性**: 定めた期間、担当者、頻度、対象範囲で実際に機能したか

手順書が存在しても実行記録がなければ、運用有効性は示せない。反対に、担当者が善意で実施していても、
責任や判定基準が未定義なら属人化し、設計上の不備となる。

## 例外とリスク受容

例外を隠すと改善機会と説明責任を失う。例外記録には、対象、理由、影響、期限、補完統制、残留リスク、
承認者、見直し日を含める。期限のない例外は恒久的な抜け穴になりやすい。

リスク受容は「対応しない」という技術担当者の判断ではなく、事業影響を負う権限者の意思決定である。

## 監査対応の流れ

1. 対象範囲、基準、評価期間、サンプル方法を確認する
2. 要求事項と統制の対応表を用意する
3. 証跡を統制ID、期間、システムごとに整理する
4. 機密情報や個人情報を必要最小限にして安全に共有する
5. 質問への回答、追加証跡、指摘事項を一元管理する
6. 指摘ごとに原因、是正、期限、オーナーを定める
7. 是正完了後に再発防止と有効性を確認する

## クラウドの責任共有

クラウド事業者の認証は、利用者の責任を代替しない。物理設備や基盤は事業者が担当しても、ID管理、
データ分類、公開設定、暗号鍵、ログ監視、バックアップ、アプリケーション脆弱性などは利用者側に残り得る。
統制ごとに、事業者、自社、委託先、顧客の責任を明記する。
